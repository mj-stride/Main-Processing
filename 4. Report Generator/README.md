# Report Generator - User Guide

Welcome to the **Report Generator** user guide. This application is designed to automate the generation of survey reports, charts, maps, and shapefiles from raw GPS trip data.

## Overview
The Report Generator takes segmented trip data (CSV files) and spatial data, and automatically produces:
- A formatted Word Document (.docx) Survey Report and Annex.
- Directional Averages (CSV)
- Speed comparison charts (PNG)
- Trip speed maps (PNG)
- ESRI Shapefiles for GIS analysis (.shp)

Everything is bundled into a single `.zip` file for easy downloading.

## Data Preparation

> [!NOTE]
> **Automated Preparation:** You do not need to manually prepare or construct this folder structure. The input for the Report Generator is the output generated by the **Main Processing**.

For the system to correctly identify and process your surveys, the uploaded files **must** follow a strict folder hierarchy (which is automatically generated by Main Processing).

## How to Use the Application

### 1. Uploading Data
You can upload data in two ways: **Folder Upload** or **ZIP Upload**.

* **Drag and Drop:** Drag your main `Region` folder (or a `.zip` archive of the folder) directly into the dashed drop zone on the screen.
* **Browse:** Click the **Browse Folder** or **Browse ZIP** buttons to manually select your files using the file explorer.

### 2. Validation
Once you select your files, the application will scan them and detect valid surveys based on the presence of the `SegmentAnalysis` folders. 
* For folders, Look at the file list below the upload box: it will display a green checkmark <i class="bi bi-check2-circle"></i> and the number of detected surveys.
* For zip files, it will display the number of uploaded zips.
* If you see a warning that "No SegmentAnalysis folders detected," your folder structure is incorrect. Please fix it and re-upload.

### 3. Generating Reports
Click the **Generate Reports** button. 
* A loading overlay will appear, showing the overall progress.
* You can monitor the real-time server logs in the dark panel to see exactly what the system is processing (e.g., "Reading CSV files...", "Generating DOCX Report...").
* **Do not close or refresh the page** while processing is ongoing.

## Output and Results
Once processing reaches 100%, the system will automatically prompt you to download a file named `Survey_Reports.zip`.

Inside the ZIP archive, the system preserves your original folder structure and injects the generated files into the respective folders:

* **DirectionalAverages/**: Contains `{Period}_DirectionalAverages.csv`
* **Graphs/**: Contains Speed Comparison Charts (`{Period}_{Direction}_SpeedPair.png`).
* **Graphs/Maps/**: Contains rendered Trip Speed Maps (`{Period}_{Direction}_TripSpeedMap.png`).
* **Shapes/shp/{Period}/**: Contains the generated ESRI Shapefiles (`.shp`, `.shx`, `.dbf`, etc.) for control-point-to-control-point segments and trip lines.
* **Survey Documents**: The root of the Vehicle Type folder will contain:
  * `{Region}_{RoadName}_{Date}_{Vehicle}_Survey_Report.docx`
  * `{Region}_{RoadName}_{Date}_{Vehicle}_Survey_Annex.docx`

## Troubleshooting

> [!WARNING]
> **Upload fails immediately:** Ensure you are uploading the parent `Region` folder or ZIP, and not just the inner CSV files.

> [!CAUTION]
> **Missing Columns Warning:** If the logs show `âš ď¸Ź SKIPPED file.csv â€” missing columns: [column_name]`, the CSV export from your data logger is incomplete. Ensure all required columns (like TravelTimeSec, DistanceM, etc.) are present.

> [!TIP]
> **No Maps Generated:** If the logs show `âš ď¸Ź Could not build trip-following CP segments`, ensure that the `Snapped` and `KM-CP Detected` folders contain the necessary `.csv` files with the correct coordinate columns (e.g. `SnappedLat`/`SnappedLon` or `latitude`/`longitude`) for the system to map the CSV data to geographic coordinates.

## 1. System Overview
The Report Generator is an **ASP.NET Core Web API** application that automates the generation of traffic survey reports from CSV and geographic data. It processes uploaded data to generate averages, visualizations (PNG charts/maps), produces GIS Shapefiles, and compiles final MS Word (.docx) reports.

### 1.1 Technology Stack
* **Framework:** .NET 8.0 (C#)
* **Web Architecture:** ASP.NET Core MVC / Web API
* **Frontend:** HTML5, CSS3, Vanilla JavaScript (Razor Pages `.cshtml`)
* **Key NuGet Packages:**
  * `CsvHelper` (33.1.0) - For parsing and writing CSV files.
  * `DocumentFormat.OpenXml` (3.5.1) - For generating DOCX reports natively without MS Office Interop.
  * `ScottPlot` (5.1.58) - For generating speed comparison bar charts.
  * `SkiaSharp` (3.119.4) - For 2D drawing and rendering the Trip Speed Maps.
  * `NetTopologySuite` (2.6.0) & `NetTopologySuite.IO.ShapeFile` - For parsing, manipulating, and exporting geographic data and ESRI Shapefiles.
  * `ProjNET` (2.1.0) - For coordinate reference system (CRS) projections.

## 2. Architecture and Workflow

The application uses an asynchronous background-processing architecture. Because report generation can take a long time, the HTTP request cannot block until completion.

### 2.1 Component Breakdown

* **`ReportController.cs`**: The API boundary.
  * `POST /api/Report/submit`: Accepts `multipart/form-data`. Buffers files into memory, enqueues the job via `ReportJobService`, and immediately returns a `jobId`.
  * `GET /api/Report/status/{jobId}`: Polled by the frontend to get state (`Queued`, `Running`, `Done`, `Failed`) and processing logs.
  * `GET /api/Report/download/{jobId}`: Streams the completed ZIP file back to the client.

* **`ReportJobService.cs` (Singleton)**: An in-memory queue and state store for jobs. Holds the `JobState`, logs array, and the final `ResultZip` byte array.
* **`ReportBackgroundService.cs` (HostedService)**: A background worker that continuously monitors the `ReportJobService` queue. When a job is found, it calls `ReportProcessingService` in a separate scope.
* **`ReportProcessingService.cs` (Scoped)**: The main orchestrator. It manages the lifecycle of generating a report: finding valid surveys, invoking parsers, invoking generators, and writing outputs to a `ZipArchive` memory stream.

### 2.2 Processing Sub-Services
* **`FolderScannerService`**: Inspects paths to identify the `Region/RoadName/Date/Vehicle/SegmentAnalysis` folder structure.
* **`CsvParserService` & `CsvExportService`**: Uses `CsvHelper` to read raw trip data and write `DirectionalAverages.csv`.
* **`DataProcessorService`**: Contains the logic to calculate `SegmentAverages` and `DirectionalAverages` (sums, speeds, delays).
* **`ChartGeneratorService`**: Uses `ScottPlot` to create side-by-side speed comparison charts (Travel vs Running Speed).
* **`TripLineLoaderService` & `SpeedSegmentService`**: Spatial logic. Reads trip linestrings and control points, maps CSV data to spatial segments.
* **`SpeedMapRenderer`**: Uses `SkiaSharp` to draw the road linestring and color-coded speed segments on a map canvas.
* **`ShapefileExportService`**: Exports the calculated segment coordinates to `.shp` files using `NetTopologySuite`.
* **`WordExportService`**: The most complex service. Uses `DocumentFormat.OpenXml` to programmatically build the Survey Report and Annex, inserting tables, text, and the previously generated PNGs.

## 3. Frontend Implementation
The frontend is a single-page interface located in `Views/Home/ReportGenerator.cshtml`. 
* **State Management**: Uses Vanilla JS.
* **File Processing**: The Drag-and-Drop zone utilizes the `webkitGetAsEntry` API to recursively walk uploaded directories on the client side. It parses file paths to provide a real-time preview of detected surveys before uploading.
* **Polling**: Once the upload completes, it sets up a `setInterval` to ping `/api/Report/status/{jobId}` every 1.5 seconds. The UI log panel is populated using regex-based milestones against the server logs to update the progress bar.

## 4. Development and Maintenance Guide

### 4.1 Running Locally
1. Ensure the .NET 8 SDK is installed.
2. Open Solution `Report Generator.sln` in Visual Studio or VS Code.
3. Run the project (`dotnet run`). The app will be available on `https://localhost:<port>`.

### 4.2 Modifying Word Templates
Currently, `WordExportService.cs` generates the DOCX entirely from scratch using the OpenXML SDK (DOM manipulation). 
> [!WARNING]
> Modifying OpenXML directly can be verbose and error-prone. If significant styling changes are required in the future, consider refactoring the app to use a **Template approach** (e.g., using `TemplateEngine.Docx` or `DotLiquid`) where a placeholder `.docx` is loaded and tokens are replaced.

### 4.3 In-Memory Limitations
> [!CAUTION]
> The application currently buffers all uploaded files, processing data, and the final ZIP archive in **RAM** (`MemoryStream`). This is fast and works well for moderate datasets. However, if users begin uploading multi-gigabyte datasets concurrently, the server may experience out-of-memory (OOM) exceptions. For high-scale production, consider refactoring `ReportController` to stream uploads directly to disk (e.g., a `Temp/` folder) and processing from disk instead of RAM.
